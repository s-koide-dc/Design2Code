# C#テスト生成機能の根本改修計画レポート

## 1. 現状の課題分析 (Updated 2026-01-27)
現在のテスト生成機能および自己修復サイクルには、以下の課題が特定されました。

1.  **自己修復の効率性**: 現在の分析・修正は「1件ずつ」実行されるため、大規模なプロジェクトで数十件のテストが失敗した場合、修復に多大なターン数を要する。
2.  **パイプラインの脆弱性**: 
    - 意図判定（IntentDetector）における正規表現の厳密すぎ・緩すぎによる誤認識。
    - セマンティック分析（SemanticAnalyzer）でのエンティティ上書きによる情報の消失（fix_id等）。
    - Windows固有の環境依存（パス区切り文字 `:` の誤認）。
3.  **破壊的修正**: スニペットレベルの修正提案をファイル全体に適用してしまい、構文エラーを引き起こす「オーバーライト問題」。

## 2. 実装済みの改善 (Fixes implemented)

### A. 基盤の堅牢化 (2026-01-27 追加)
- **意図判定の最適化**: `APPLY_CODE_FIX` の正規表現を非貪欲 (`.*?`) に変更し、抽出精度を向上。承認待ち状態での `AGREE` 優先判定を強化。
- **エンティティ保護**: `SemanticAnalyzer` でのエンティティ抽出を「上書き」から「マージ」に変更し、`fix_id` 等の重要なコンテキストを維持。
- **Windowsパス対応**: 絶対パスに含まれるドライブレターの `:` を行番号と誤認しないよう、正規表現によるパス解決を導入。
- **ロギング修正**: `LogManager` のメソッド不一致（`log_error` vs `log_event`）を解消。

### B. 部分置換ロジックの導入
- **行単位修正**: `test_self_healing` タイプの修正において、履歴から失敗行番号を特定し、該当行のみを置換するロジックを実装。これにより、既存のテストコードを破壊せず安全にアサーションを更新可能に。

### C. 実行フィードバックループの強化
- **ビルド先行チェック**: テスト実行前に `dotnet clean` および `build` を実施。エラー時は原因（stdout/stderr両方）をユーザーへ即座に通知。

## 3. 次の目標: 一括自己修復の実現 (Next Goals - Phase 5) - 実装済み

1.  **一括失敗分析 (Bulk Analysis)**:
    - 1回の指示で、失敗しているすべてのテストケースをスキャン。
    - 各失敗原因（期待値不一致、Null、例外等）を個別に分析し、修正案のリストを生成。
2.  **一括修正適用 (Bulk Apply)**:
    - 生成された全修正案を、単一のアクションで順次適用。
    - 全適用後に一括でビルド検証を行い、整合性を確認。
3.  **自律的再試行 (Autonomous Retries)**:
    - 修正→テスト→残った失敗を再分析→再修正、というループをユーザーの介入なしに最大 N 回繰り返す。

## 4. 新たな挑戦: テストデータ（Arrange）の自己修復 (Next Goals - Phase 6) - Completed & Integrated

一括修正の導入により「ガード節の挿入」や「期待値の更新」は自動化されましたが、依然として「テスト対象に渡すデータが不適切でエラーになる」という課題が残っています。これを解決するため、分析の矛先を「出口」から「入口」へと転換しました。

1.  **深層根本原因分析 (Deep Root Cause Analysis)**:
    - **(実装済)** スタックトレースの行番号前後をスキャンし、実際にNREの原因となったメソッド呼び出しを特定する頑健なパースロジックを実装。
2.  **テスト準備コード（Arrange）の動的生成**:
    - **(実装済)** 特定された依存関係の型と戻り値を解析し、テストコードの適切な場所（Actフェーズの直前）にモック設定を自動挿入する機能を実装。
3.  **ドメイン特化型ダミーデータ生成の統合**:
    - **(実装済)** `DummyDataFactory` により、クラスのプロパティ初期化（`new DataItem { Value = "test" }`）まで考慮した高度なダミーデータを生成。

## 5. 次の目標: 精度向上とシナリオベースのテスト生成 (Next Goals - Phase 7)



これまでのフェーズで「一括修復」や「Arrangeデータの自動生成」といった基盤が整いました。次のステップでは、修復の精度を極限まで高め、より網羅的なテスト生成を目指します。



1.  **文脈適応型修正 (Context-Aware Fixes)**:

    - 修正案を生成する際、対象ファイルのスコープ内にある変数やメソッドを静的に解析し、未定義の変数を参照するような「壊れた修正案」の生成を防止。

2.  **分岐網羅型テスト生成 (Logic-Driven Scenarios)**:

    - 静的解析（Roslyn/AST）により実装内の `if`, `switch`, `try-catch` 等の分岐を特定。単なる型ベースの推論ではなく、ロジックを通過するための具体的な入力値を逆算してテストケースを生成。

3.  **依存関係の連鎖修復 (Multi-file Propagation)**:

    - インターフェースの変更が複数のクラスやテストに影響する場合、依存関係グラフを辿って関連する全ファイルを一貫性を保ったまま一括修正。

4.  **修正案の自己修正ループ (Meta-Healing)**:

    - AIが提案した修正を適用してビルドエラーになった場合、そのエラー内容自体を再度分析し、修正案を自律的に修正して再試行する機能を強化。



## 6. 将来の目的: 完全自律型TDDサイクルの確立



本プロジェクトの最終目標は、**「仕様（目標）から動作するコードとテストを自律生成する」**ことです。



*   **決定論的推論の極致**: 確率的な生成に頼らず、Roslynによる静的解析と実際の実行結果（実行時動的情報）を組み合わせ、100%ビルド可能で、仕様を満たすコードを導き出す。

*   **「仕様書＝コード」の実現**: 人間が `.design.md` に目的を書けば、AIがテストを書き、そのテストを通すための最小実装を書き、カバレッジ100%になるまで自律的に品質を高め続ける環境を構築する。
