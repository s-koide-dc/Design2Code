# IR Validator Module Design Document

## 1. Purpose
The `IRValidator` performs static analysis on the Intermediate Representation (IR) tree generated by `IRGenerator`. It ensures logical consistency, data flow validity (e.g., variables are defined before use), and type compatibility before the `CodeSynthesizer` attempts to generate code.

## 2. Structured Specification

### Input
- `ir_tree` (Dict): The logical tree generated by `IRGenerator`.
- `entity_schema` (Dict): Project-wide entity definitions.
- `type_system` (TypeSystem): The project's type compatibility logic.

### Output
- `validation_result` (Dict):
    - `is_valid` (bool): True if no critical logical errors are found.
    - `errors` (List[str]): Descriptions of found issues.
    - `warnings` (List[str]): Non-critical suggestions.

### Core Logic
1.  **Symbol Table Tracking**:
    -   Recursively traverse the IR tree.
    -   Maintain a scope-aware symbol table (available types and variables).
2.  **Unreachable Node Detection**:
    -   Identify blocks that can never be executed (e.g., empty `if` bodies).
3.  **Data Flow Validation**:
    -   Ensure that for each `ACTION` or `TRANSFORM` node, the `target_entity` or implied input is available in the current scope.
    -   Example: If a step says "Save users" but no previous step "Got users", mark as an error.
4.  **Type Consistency (Heuristic)**:
    -   Check if the output type of a node is compatible with the expected input type of the next node.
5.  **Loop/Condition Integrity**:
    -   Check for balanced `LOOP` and `CONDITION` blocks.

### Test Cases
- **Happy Path**: A valid sequence of "Get" then "Process" then "Save".
- **Undefined Variable**: "Save users" without a prior "Get users".
- **Empty Block**: A `CONDITION` node with an empty `body`.
- **Incompatible Flow**: "Get count" (int) followed by "Filter by name" (string property check).

## 3. Dependencies
- `TypeSystem`: For type compatibility checks.
- `entity_schema.json`: For entity metadata.
